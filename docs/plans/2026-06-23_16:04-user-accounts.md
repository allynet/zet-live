# Plan: User Accounts (Social Login + Settings Sync)

> ## UPDATE (post-implementation): sessions are BEARER TOKENS, not cookies
>
> The sections below originally specified **httpOnly cookie** sessions. During
> development this turned out to be a major source of friction: the frontend
> (Vite) and backend run on **different origins** in dev (`saturn.ji0.li:5173`
> ↔ `:9011`), and httpOnly cookies are host-scoped + `SameSite`-restricted, so
> they refused to travel cross-origin. Every issue encountered (state cookie,
> session cookie, `postMessage` origin, CORS credentials, `localhost` vs
> `saturn.ji0.li` host matching) traced back to this.
>
> **Decision: switch to long-lived bearer tokens stored in `localStorage`.**
>
> - The OAuth Authorization Code + PKCE flow is unchanged and correct.
> - The session is still DB-backed (`user_sessions`, SHA-256 hash of the token).
> - On login, the callback delivers the raw token to the opener via
>   `postMessage` (with `targetOrigin` = the opener's origin, sent in
>   `/start?origin=…`), **not** via a `Set-Cookie`. The client stores it in
>   `localStorage` and sends it as `Authorization: Bearer …`.
> - The short-lived `oauth_state` cookie **remains** — it lives entirely on the
>   backend origin inside the popup (`/start` sets it, `/callback` reads it), so
>   it has zero cross-origin issues, and it provides the OAuth CSRF binding.
> - `CORS_ORIGINS` / credentials support was **reverted** — not needed, since
>   bearer tokens use a header (not cookies) and the existing permissive CORS
>   already allows it (no credentials mode).
> - **Security tradeoff (accepted):** a token in `localStorage` is readable by
>   XSS, unlike an httpOnly cookie. For an account that only stores map
>   settings, this is acceptable — and as the user noted, an XSS that can read
>   `localStorage` can equally drive the page to do whatever it wants, cookie or
>   not. Tokens are revocable (logout deletes the session row).
>
> The detailed sections below are left as historical context; where they
> describe cookie mechanics, the implementation uses bearer tokens instead.

## Goal

Add user accounts to ZET Live, authenticated exclusively via social login (OAuth2).
Initial provider scope: **Google** and **Microsoft** (Facebook and Apple follow the same
pattern later). Users may link multiple providers to a single account. The accounts are
initially used only to **store and sync user settings** across devices — nothing else.

Key UX/feature requirements agreed with the user:

- Social logins only (no passwords, no email/password registration).
- Multiple providers linkable to one account.
- Accounts used for settings storage.
- Auth must be **versioned** under `/api/v1/auth/*` (same version flag as the rest of the API).
- The frontend must **dynamically discover** which auth providers the backend has configured
  and only render login buttons for those. If no providers are configured, the
  login/account button is hidden entirely.

## Design Decisions (and rationale)

### 1. Self-hosted in Rust (not a managed auth service)

**Decision:** Implement OAuth2 in the backend. Do NOT use Clerk / Auth0 / Supabase Auth /
Logto.

**Rationale:** The project is a self-hosted monolith (Docker → scratch, SQLite, single Rust
binary). A managed service would add an external dependency, potential per-user cost, and a
second system to operate. Self-hosting keeps everything in the monolith with full control
and works offline/locally. The implementation surface for two providers is modest.

### 2. OAuth2 + userinfo endpoint (not OIDC / `openidconnect`)

**Decision:** Use the `oauth2` crate with the **Authorization Code flow + PKCE**, then call
the provider's **userinfo endpoint** to obtain the user's stable identifier. Do NOT use the
`openidconnect` crate (ID-token/JWKS verification).

**Rationale:**
- `openidconnect` does strict issuer matching, which is painful for Microsoft's "common"
  tenant (the issuer in the ID token is tenant-specific). Avoiding it sidesteps JWKS
  fetching, issuer/tenant edge cases, and extra complexity.
- Both Google (`sub` from `https://openidconnect.googleapis.com/v1/userinfo`) and Microsoft
  (`id` from Graph `/me`) expose stable, per-application user identifiers via their userinfo
  endpoints over HTTPS. For a settings-only app this is the standard, sufficient trust model.
- PKCE protects the authorization-code exchange even though the client secret is server-side
  (defense in depth; also required/simpler for some providers).

### 3. Server-side sessions in SQLite, httpOnly cookies (not JWT, not localStorage tokens)

**Decision:** Maintain a `user_sessions` table. The session token is 32 cryptographically
random bytes; store only its **SHA-256 hash** in the DB. Deliver the raw token via a
`session` cookie that is `HttpOnly`, `Secure` (in production), `SameSite=Lax`, `Path=/`.

**Rationale:**
- The frontend is served **same-origin** by the backend, so httpOnly cookies work for both
  REST and the WS upgrade with zero JS token handling — and they're XSS-resistant (no token
  in JS to steal).
- Server-side sessions are **revocable** (delete the row), unlike stateless JWTs.
- Hashing the token in the DB means a DB read-leak does not immediately equal session
  hijack.
- No token refresh / expiry-juggling logic needed.
- `SameSite=Lax` blocks cross-origin state-changing POSTs (handles CSRF for our own POST
  endpoints). The OAuth callback GET is protected by the `state` parameter + a signed
  short-lived `oauth_state` cookie. (See Security.)

### 4. Cookie model over bearer tokens (no Authorization header plumbing)

**Decision:** Authenticate via the `session` cookie, read by a `CurrentUser` Axum
`FromRequestParts` extractor. The frontend adds `credentials: "include"` to `apiFetch`
(harmless same-origin, required if `VITE_API_URL` is ever cross-origin in dev).

**Rationale:** Minimal change to the frontend (`apiFetch` is the single fetch wrapper).
WebSocket auth is automatic (cookies on the upgrade). No token storage in Zustand/localStorage.

### 5. Explicit account linking only (no email-based auto-merge)

**Decision:** A `users` row owns N `user_oauth_identities` rows.
- First login with any provider → create a `users` row + an identity row.
- **Linking** a second provider requires an **existing session** (`?link=1` on the start
  URL); the new identity is attached to the current user.
- Logins are keyed on `(provider, provider_subject)`. If a provider identity exists, the
  corresponding user is reused. **Emails never trigger auto-merge.**

**Rationale:** Email-based auto-merge is an account-takeover vector (an attacker who controls
a provider account with a matching email could gain access to an existing user). Explicit
linking is safe and matches the user's requirement ("users should be able to link multiple
providers"). Edge case: two providers with the same email but never explicitly linked stay
as separate accounts — acceptable for a settings app.

### 6. Popup OAuth UX (not full-page redirect)

**Decision:** The frontend opens a centered popup to `/api/v1/auth/{provider}/start`. After
the provider redirects back to the backend callback, the backend returns a tiny HTML page
that `postMessage`s the opener window and closes itself. The opener then fetches
`/auth/me` and updates the UI.

**Rationale:** The map app stays mounted (no remount, no lost in-memory state). Slightly
more code than a full-page redirect but a much better SPA UX. Same-origin makes
`window.opener` + `postMessage` straightforward.

### 7. Last-write-wins settings sync with first-login push

**Decision:** When authenticated, the **server is the source of truth** and the local
localStorage mirror is kept in sync:
- **On becoming authenticated** (login, or app load while authenticated): `GET /settings`.
  - If `404` (no server settings yet) → `PUT` the current local settings (first-login push,
    so a user who customized before logging in doesn't lose them).
  - If exists → **replace local settings** with the server's (server wins for returning
    devices).
- **On any local setting change while authenticated** → debounced `PUT /settings`
  (last-write-wins across devices).
- **On logout** → keep the current settings in localStorage untouched.

**Rationale:** Predictable, simple, no merge UI. Multi-device staleness resolves on the next
login/load. Returning users always pull latest server settings on app load.

### 8. Versioned auth routes — `/api/v1/auth/*`

**Decision:** All auth endpoints live in the existing v1 router alongside `/vehicles`,
`/schedule/*`, etc. No unversioned router.

**Rationale:** Explicit user requirement. Keeps auth on the same versioning contract as the
rest of the API.

### 9. Backend "capabilities" discovery — `GET /api/v1/capabilities`

**Decision:** Add a dedicated `GET /api/v1/capabilities` endpoint that reports what the
backend supports:

```json
{
  "auth": {
    "providers": [
      { "id": "google", "name": "Google" },
      { "id": "microsoft", "name": "Microsoft" }
    ]
  }
}
```

The configured-providers map (built from env at startup) is the **single source of truth**
for three things:
1. Which `/{provider}/start` + `/{provider}/callback` routes are registered (unconfigured
   providers are not registered → 404).
2. The contents of the `capabilities` response.
3. Valid values for `DELETE /auth/identities/{provider}`.

`/auth/me`, `/auth/logout`, `/auth/identities/{provider}`, `/settings` are always registered
(they return `401` without a session; harmless when no providers are configured since nobody
can ever log in).

**Rationale:** Explicit user requirement that the frontend be data-driven: enabling/disabling
a provider is a pure backend config change (`AUTH_GOOGLE_*` set/unset) and the frontend
adapts automatically — the account button appears/disappears and only configured provider
buttons render. A dedicated endpoint (rather than merging into `/version`) keeps build-info
and runtime-config concerns separate and is extensible for future feature flags.

**Alternative considered (rejected):** merge capabilities into `GET /api/v1/version` (which
is already polled by `use-version-check.ts`). Rejected to keep concerns separate; the extra
request is negligible and cacheable.

### 10. Settings stored as an opaque JSON blob

**Decision:** `user_settings.settings` is a JSON `BLOB`. The backend does not interpret the
shape; the frontend owns the schema and validates it with Zod on read.

**Rationale:** The frontend settings object is small and frontend-defined. Server-side
schema enforcement would couple the two and is unnecessary for a trusted-client settings
sync. Keeps the backend generic.

### 11. Provider selection — Google + Microsoft first

**Decision:** Implement Google and Microsoft in the initial scope. Facebook and Apple use the
identical pattern and can be added later by (a) adding a `Provider` entry, (b) registering
an OAuth app, (c) dropping an icon in `frontend/src/utils/provider-icons.tsx`.

**Rationale:** Explicit user choice. Two providers is enough to prove multi-provider + linking
works end-to-end while keeping scope bounded.

## OAuth Flow (detailed, Google example)

1. User clicks "Continue with Google" in the frontend modal.
2. Frontend `openOAuthPopup("google")` opens a centered popup to
   `GET /api/v1/auth/google/start` (or `…?link=1` when already authenticated and linking).
3. Backend generates a PKCE challenge and a random `state`, stores
   `state + pkce_verifier + link_flag + provider` in a **short-lived signed `oauth_state`
   cookie** (signed with `SESSION_SECRET`), and `302`s to Google's authorization URL with
   `code_challenge`, `state`, and scopes `openid email profile`.
4. User authenticates at Google; Google `302`s the popup to
   `GET /api/v1/auth/google/callback?code=…&state=…`.
5. Backend validates the `state` against the `oauth_state` cookie, exchanges the code for
   tokens using the stored PKCE verifier, then calls Google's userinfo endpoint →
   `{ subject, email, name, avatar }`.
6. Upsert:
   - Look up `user_oauth_identities` by `(provider="google", provider_subject=subject)`.
   - **Login (no session):** if found, reuse its `user_id`; else create a new `users` row +
     the identity.
   - **Link (`?link=1` + valid existing session):** if the identity already belongs to
     **another** user → error. Else attach the identity to the current user.
   - Update `provider_email/name/avatar` on the identity and the user's `display_name/email/
     avatar_url` from this provider (last-provider-used wins).
7. Create a session (32 random bytes → store SHA-256 hash + user_id + expiry + ip + ua),
   set the `session` cookie, clear the `oauth_state` cookie.
8. Return a tiny HTML page that does `window.opener.postMessage({type:"oauth-callback",
   ok:true, provider:"google"}, <app origin>)` then `window.close()`.
9. Opener (which listened with an origin check) receives the message → fetches
   `GET /api/v1/auth/me` → updates the auth store. On failure, `postMessage` carries the
   error and the modal surfaces it.

Microsoft is identical with Microsoft Identity Platform v2.0 (`common` tenant) endpoints and
Graph `/me` for userinfo.

## Database Schema

New migration: `backend/migrations/<ts>_user_accounts.{up,down}.sql` (timestamp >
`20260617120000`). All tables `STRICT`.

```sql
CREATE TABLE users (
    id          TEXT PRIMARY KEY,        -- ULID
    display_name TEXT,
    email       TEXT,
    avatar_url  TEXT,
    created_at  TEXT NOT NULL,           -- ISO 8601 (jiff)
    updated_at  TEXT NOT NULL
) STRICT;

CREATE TABLE user_oauth_identities (
    id                   TEXT PRIMARY KEY,        -- ULID
    user_id              TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    provider             TEXT NOT NULL,           -- 'google' | 'microsoft' | ...
    provider_subject     TEXT NOT NULL,           -- provider's stable user id (sub / id)
    provider_email       TEXT,
    provider_display_name TEXT,
    provider_avatar_url  TEXT,
    created_at           TEXT NOT NULL,
    updated_at           TEXT NOT NULL,
    UNIQUE (provider, provider_subject)
) STRICT;
CREATE INDEX idx_user_oauth_identities__user_id ON user_oauth_identities (user_id);

CREATE TABLE user_sessions (
    token_hash TEXT PRIMARY KEY,          -- SHA-256 hex of the raw session token
    user_id    TEXT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    expires_at TEXT NOT NULL,
    created_at TEXT NOT NULL,
    ip         TEXT,
    user_agent TEXT
) STRICT;
CREATE INDEX idx_user_sessions__user_id ON user_sessions (user_id);

CREATE TABLE user_settings (
    user_id    TEXT PRIMARY KEY REFERENCES users(id) ON DELETE CASCADE,
    settings   BLOB NOT NULL,             -- JSON blob, frontend-owned schema
    updated_at TEXT NOT NULL
) STRICT;
```

`.down.sql` drops the four tables in reverse dependency order.

**Required after writing new `query!` macros:** regenerate the `.sqlx/` offline cache with
`DATABASE_URL=sqlite:./dev/db/db.sqlite cargo sqlx prepare` (run from `backend/`), or the
offline build fails.

## API Endpoints

All under `/api/v1`.

| Method | Path | Auth | Purpose |
|--------|------|------|---------|
| GET | `/capabilities` | none | `{ auth: { providers: [{id,name}] } }` (data-driven UI) |
| GET | `/auth/{provider}/start` | none (reads session if `?link=1`) | PKCE + state cookie → 302 to provider |
| GET | `/auth/{provider}/callback` | none | Validate state, exchange code, upsert user/identity, create session, return popup-closer HTML |
| POST | `/auth/logout` | session | Destroy session row, clear `session` cookie |
| GET | `/auth/me` | none (returns 401 if anon) | `{ user: {id, display_name, email, avatar_url}, identities: [{provider, provider_email, …}] }` |
| DELETE | `/auth/identities/{provider}` | session | Unlink provider (refuse if it's the last identity) |
| GET | `/settings` | **session** | Current user's settings JSON blob (404 if none) |
| PUT | `/settings` | **session** | Upsert settings JSON blob |

`{provider}` path param is validated against the configured-providers map → 404 if unknown.

### Auth store + hook

- `src/auth-store.ts` (in-memory only — cookies persist auth, not localStorage):
  `{ user: User|null, identities: Identity[], status: 'loading'|'authenticated'|'anonymous' }`.
- `src/hooks/use-auth.ts`:
  - On mount: `GET /auth/me` → set status (`authenticated`/`anonymous`).
  - `loginWith(provider, {link})`: `openOAuthPopup(...)` then re-fetch `/auth/me`.
  - `logout()`: `POST /auth/logout`, clear store → anonymous.
  - `unlink(provider)`: `DELETE /auth/identities/{provider}`, re-fetch `/auth/me`.

### OAuth popup util

`src/utils/oauth-popup.ts` — `openOAuthPopup(provider, {link})`:
- Compute URL `${API_URL}/v1/auth/${provider}/start` (+ `?link=1`).
- Open a centered popup (`width=500,height=650`).
- Add a `message` listener that checks `event.origin === appOrigin` and resolves on
  `{type:'oauth-callback', ok:true}` / rejects on `ok:false`, with a timeout fallback
  (reject) and a popup-close detection fallback.

### Settings sync

`src/hooks/use-settings-sync.ts` — reactive, mirrors `subscribeWithSelector` patterns:
- Subscribe to auth `status`. On `authenticated`: `GET /settings`; 404 → `PUT` local; else
  replace local with server.
- Subscribe to settings changes while `authenticated`: debounce ~500ms → `PUT /settings`.

### Auth button + modal

- `src/components/auth-button.tsx`: round 36px button (same style as Settings/Feedback
  buttons) mounted in the top-right column. Renders only if
  `capabilitiesStore.authProviders.length > 0`. Shows avatar (if available) when
  authenticated, a generic user icon otherwise.
- `src/components/auth-modal.tsx`: modeled on `feedback-modal.tsx`/`settings-modal.tsx`.
  - **Anonymous:** list of provider buttons **data-driven from `authProviders`**, each with
    its icon from `provider-icons.tsx`. Clicking calls `loginWith(provider)`.
  - **Authenticated:** show display name/email/avatar, the list of linked providers, a
    "Link another provider" row (buttons for providers not yet linked), "Unlink" buttons
    (disabled when it's the last identity), and "Sign out".
- `src/utils/provider-icons.tsx`: `{ google: <GoogleSvg/>, microsoft: <MicrosoftSvg/> }` +
  display names. Adding a future provider = drop in an entry.

### App wiring

In `src/app.tsx`: call `useCapabilities()`, `useAuth()`, `useSettingsSync()` near the top;
render `<AuthButton />` + `<AuthModal />` alongside the existing buttons/modals.

## Security Considerations

| Concern | Handling |
|---------|----------|
| XSS token theft | Mitigated: no token in JS; httpOnly cookie. |
| CSRF on POST endpoints | `SameSite=Lax` cookie blocks cross-origin state-changing requests. |
| CSRF on OAuth callback (GET) | `state` param validated against the signed `oauth_state` cookie. PKCE on every flow. |
| Session hijack via DB leak | Only the SHA-256 hash of the token is stored; raw token lives only in the cookie. |
| Session fixation | Fresh random token on every login. |
| Account takeover via email | No email-based auto-merge; linking is explicit and requires an existing session. |
| Popup origin spoofing | Opener verifies `event.origin === appOrigin` before acting on the postMessage. |
| Unlinking last identity | Refused by the backend (user would lose all access); returns an error. |
| Open redirect | Redirect URIs are fixed per-provider config (`${APP_URL}/api/v1/auth/{provider}/callback`), not user-controlled. |

## Configuration & Prerequisites (manual, outside code)

- Register OAuth apps:
  - **Google Cloud Console** → OAuth client (Web), add authorized redirect URI
    `${APP_URL}/api/v1/auth/google/callback`, copy client ID/secret.
  - **Microsoft Entra / Azure portal** → app registration, Web platform redirect URI
    `${APP_URL}/api/v1/auth/microsoft/callback`, create client secret.
- Set env vars in `.env` / deployment:
  ```
  APP_URL=https://zet-live.example.com
  SESSION_SECRET=<random 32+ bytes>
  SESSION_MAX_AGE_SECS=2592000
  AUTH_GOOGLE_CLIENT_ID=...
  AUTH_GOOGLE_CLIENT_SECRET=...
  AUTH_MICROSOFT_CLIENT_ID=...
  AUTH_MICROSOFT_CLIENT_SECRET=...
  ```
- **Dev note:** if running the frontend on `:5173` and backend on `:9011` separately, the
  OAuth callback (full navigation in the popup) and the session cookie must be same-origin.
  Either rely on the Vite proxy for `/api` (verify `vite.config.ts` proxies `/api` →
  `:9011`) or run everything through the backend in dev. Verify during implementation.

## Scope Boundaries

- **In:** Google + Microsoft login; account linking/unlinking; sessions/logout; per-user
  settings sync (last-write-wins); popup UX; capabilities discovery; account modal UI.
- **Out (future, same patterns):** Facebook and Apple providers; account deletion;
  email-based account merging; per-user WS personalization; password/2FA (n/a — social only).

## Risks & Mitigations

| Risk | Mitigation |
|------|------------|
| `.sqlx/` offline cache out of date after adding `query!` macros | Regenerate with `cargo sqlx prepare` (live `DATABASE_URL`) before building. |
| Microsoft `common` tenant issuer mismatch (if OIDC were used) | Avoided entirely by using the userinfo endpoint instead of `openidconnect`. |
| Popup blocked / closed early by user | `openOAuthPopup` detects close + timeout; rejects the promise; modal surfaces a retry. |
| Cross-origin in dev breaks cookies | Verify Vite `/api` proxy; document running through backend as fallback. |
| `corsLayer::very_permissive` + credentials | `Allow-Origin: *` makes browsers refuse credentials cross-origin; same-origin unaffected. No change needed; documented. |
| Provider userinfo shape drift | Normalize via a single `ProviderUserInfo`; fail closed (error → no session) on unexpected response. |
| Two devices race on settings | Last-write-wins; resolves on next load. Acceptable for a settings app. |
| Unlink leaving a user with no access | Backend refuses unlinking the last identity. |
| `SESSION_SECRET` unset in prod | Require at startup when ≥1 provider configured; otherwise sessions can't be signed. |
